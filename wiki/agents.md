# MDBlog — Agent Instructions (Wiki)

## Project Overview

MDBlog is a lightweight, flat-file blog engine written in **Go 1.26**. Posts are Markdown files with YAML-style front matter. There is no database, no build step, and no JavaScript framework. Keep it that way.

## Architecture

```
cmd/
  mdblog/           # CLI entry point: serve | build-index | build-feed | build-sitemap | render | request | version
  lambda/           # AWS Lambda entry point (reads templates/assets from disk)
  lambda-embed/     # AWS Lambda entry point (templates+assets embedded in binary)
internal/
  blog/             # Core domain: post scanning, slug resolution, pagination, menu, pages
  buildindex/       # Build-time index generator (writes posts/posts.index.json)
  buildfeed/        # Build-time RSS feed generator (writes feed.xml from the index)
  buildsitemap/     # Build-time SEO generator (writes sitemap.xml and robots.txt)
  config/           # Config loader — parses config.toml into Config struct
  markdown/         # Front matter parser + Goldmark renderer
  render/           # CLI render subcommand (renders a post to a standalone HTML file)
  server/           # net/http handler: routing, template execution, gzip, CSP
    handler.go      # HTTP routing and page rendering
    seo.go          # JSON-LD structured data + sitemap/robots serving
templates/          # Go html/template files (*.html)
assets/
  css/              # CSS themes
  js/               # Optional per-post JavaScript files
posts/              # All blog post content lives here
  index.md          # Optional landing page blurb (shown above category cards)
  *.md              # Root-level (uncategorized) posts
  srbyte/           # Category folder
  substack/         # Category folder
pages/              # Standalone pages (e.g. about.md) — no date, no category, no tags
posts/posts.index.json  # Pre-built post metadata index (generated by `make build-index`)
feed.xml            # Pre-built RSS 2.0 feed (generated by `make build-feed`, baked into Docker image)
sitemap.xml         # Pre-built sitemap (generated by `make build-sitemap`, baked into Docker image)
robots.txt          # Pre-built robots.txt (generated by `make build-sitemap`, baked into Docker image)
embed.go            # go:embed declarations; exposes EmbeddedTemplates() / EmbeddedAssets()
config.toml         # All runtime configuration (single source of truth)
Dockerfile          # Production Lambda image: builds lambda-embed binary (templates/assets embedded)
Dockerfile.debug    # Debug variant: builds lambda & mdblog, copies content/pages/templates/assets
docker-compose.yml  # Local dev only — runs /mdblog serve on port 8080 (via Dockerfile.debug)
Makefile            # Developer targets: help, serve, build, test, new-post, docker-*
```

**Deployment model:** The production artifact is a Docker container image pushed to a registry (e.g. ghcr.io) and deployed as an **AWS Lambda container image function** behind API Gateway. The `cmd/lambda` binary uses [algnhsa](https://github.com/akrylysov/algnhsa) to translate API Gateway HTTP events into standard `http.Handler` calls. Posts and all static assets are **baked into the image** — there is no external content source or database.

**Embed variant:** The default `Dockerfile` builds a single binary (`cmd/lambda-embed`) which has `templates/` and `assets/` embedded via `go:embed`. This is the production deployment image built in the CI. The `Dockerfile.debug` builds the standard binaries and copies `templates/` and `assets/` to disk for local development preview via `docker-compose.yml`.

**Continuous Deployment:** A GitHub Action is configured to automatically build and release a new container image to GHCR whenever a Markdown (`.md`) file inside the `posts/` folder is pushed to `master`. A chained workflow then automatically deploys the latest image to AWS ECR and updates the Lambda function.

`docker-compose.yml` is for local development only and is not used in production.

## Configuration

All settings live in `config.toml` (TOML format). They are parsed by `internal/config` into a `Config` struct. Key fields:

| Key | Type | Purpose |
|-----|------|---------|
| `blog_name` | string | Site title |
| `author_name` | string | Default post author (used by `make new-post`) |
| `lang` | string | HTML `lang` attribute (default `"en"`) |
| `header_content` | string | Landing page subtitle |
| `footer_content` | string | Footer text (Markdown supported) |
| `posts_per_page` | int | Pagination size (default 10) |
| `excerpt_length` | int | Max excerpt length in characters (default 200) |
| `show_uncategorized` | bool | Include root-level posts in listings |
| `uncategorized_label` | string | Display name for uncategorized posts |
| `posts_dir` | string | Root posts directory (default `"posts"`) |
| `pages_dir` | string | Standalone pages directory (default `"pages"`) |
| `post_index_file` | string | Path to pre-built metadata index |
| `date_format` | string | Go time layout for dates (default `"2006-01-02"`) |
| `css_theme` | string | Path to active CSS theme |
| `show_render_time` | bool | Append render-time HTML comment |
| `[csp]` | struct | `enabled` + `header` string |
| `[feed]` | struct | `enabled`, `title`, `description`, `base_url` (req when enabled), `max_items` (default 50), `output_file` (default `"feed.xml"`) |
| `[sitemap]` | struct | `enabled`, `output_file` (default `"sitemap.xml"`), `robots_file` (default `"robots.txt"`), `changefreq_home/category/post`, `priority_home/category/post` |
| `[[menu_links]]` | array | Static nav links: `label` + `url` |
| `[categories.<slug>]` | map | Category definitions (see below) |
| `[labels]` | struct | All user-visible UI strings |

When adding a new config key, add it to `config.toml` and the `Config` struct in `internal/config/config.go`, then consume it via the `*config.Config` pointer passed to handlers. Never hardcode values in templates or handlers.

## Post Format

```yaml
---
title: My Post Title
date: YYYY-MM-DD
author: Name
tags: tag1, tag2
description: Optional meta description
js: optional-script.js   # loaded from assets/js/
---

Markdown body here (GitHub Flavoured Markdown + footnotes).
```

File naming convention: `YYYY-MM-DD-slug-with-hyphens.md`.
*Note on Slugs:* The post slug is derived directly from the filename (e.g. `2026-06-03-my-post.md` resolves to the slug `2026-06-03-my-post`). If a clean slug without a date prefix is desired (e.g., `/post?slug=mdblog`), name the file without the date prefix (e.g. `mdblog.md`) and ensure you explicitly define the `date` key in the front matter.
Category posts live under `posts/<category-folder>/`.

## Internal Packages

### `internal/config`

Loads and validates `config.toml`. Exported functions:
- `Load(path string) (*Config, error)` — parse and apply defaults
- `MustLoad(path string) *Config` — panics on error (used by entry points)

### `internal/markdown`

Front matter extractor and Goldmark-based renderer.
- `Parse(content string) ParsedDoc` — extract front matter + render body to HTML
- `ParseMetaOnly(content string) MetaDoc` — extract front matter + return raw body (no rendering; used by `buildindex`)
- Goldmark is configured with GFM extensions (tables, strikethrough, task lists), footnotes, auto heading IDs, hard wraps, and **safe mode** (no raw HTML passthrough — do not add `html.WithUnsafe()`).

### `internal/buildindex`

Scans all post directories, calls `ParseMetaOnly` for each, and writes `posts/posts.index.json` atomically. Full Goldmark rendering is **never invoked** during index generation.
- `Build(cfg *config.Config) error`

### `internal/blog`

Core domain logic. The `Blog` struct is the main service object:
- `New(cfg *config.Config) *Blog`
- `GetPosts(category string, page int) (PostList, error)` — paginates from the JSON index
- `SearchPosts(query string, page int) (PostList, error)` — full-text search via the JSON index
- `GetPostBySlug(slug, category string) (*Post, error)` — reads + renders a single post
- `GetPage(slug string) *Page` — reads + renders a standalone page from `PagesDir`; path-traversal-safe
- `GetCategories() map[string]*CategoryInfo`
- `GetCategoryBySlug(slug string) (*CategoryInfo, bool)`
- `GetMenu() []MenuLink` — returns flat links (`[[menu_links]]`, pinned) + multiple dropdown `MenuLink` entries configured in `[[menu.dropdowns]]`
- `GetVersionInfo() VersionInfo`

### `internal/server`

`net/http` handler. Key details:
- `New(cfg *config.Config, b *blog.Blog) *Handler` — loads templates, returns the handler
- `TemplateFS` and `AssetsFS` package-level variables — override in embed builds
- Routes: `/` (home/category/search), `/feed.xml` (RSS), `/feed` (human-readable feed page), `/post` (single post), `/page` (standalone page), `/sitemap.xml`, `/robots.txt`, `/assets/*` (static files)
- Gzip compression via `compress/gzip` (when `Accept-Encoding: gzip` is present)
- CSP header injected per request when `csp.enabled = true`
- Templates use `html/template` with custom functions: `safeHTML`, `formatDate`, `urlquery`, `postPreviewData`, `defaultStr`
- **`handler.go`**: routing, template rendering, HTTP helpers
- **`seo.go`**: JSON-LD structured data builders (`buildWebSiteJSONLD`, `buildWebPageJSONLD`, `buildArticleJSONLD`) and HTTP handlers for `/sitemap.xml` and `/robots.txt` (serves pre-built files, falls back to dynamic generation during local dev)

### `internal/buildfeed`

Build-time RSS 2.0 feed generator.
- `Build(cfg *config.Config) error` — reads `posts.index.json`, writes `feed.xml` atomically
- No Goldmark rendering — excerpts come from the pre-built index
- No external dependencies — uses `encoding/xml` from the standard library
- Called by `make build-feed`; must be run after `make build-index`
- `feed.enabled = true` in `config.toml` requires `feed.base_url` to be set (returns error otherwise)

### `internal/buildsitemap`

Build-time SEO file generator.
- `Build(cfg *config.Config) error` — reads `posts.index.json`, writes `sitemap.xml` and `robots.txt` atomically
- No Goldmark rendering — reads only slug, date, and category_slug from the index
- Called by `make build-sitemap`; must be run after `make build-index`
- `sitemap.enabled = true` requires `feed.base_url` to be set
- `changefreq` and `priority` for home/category/post pages are fully configurable via `[sitemap]` in `config.toml`
- The HTTP handler in `server/seo.go` serves the pre-built files in production; falls back to a live dynamically-generated response for local dev (so `make serve` works without running `make build-sitemap`)

### `internal/render`

CLI subcommand for rendering a single post to a standalone HTML file in a `render/` output directory (used by `make render`). It also supports the `request` command (used by `make request URL="..."`), which simulates a GET request against any site endpoint (posts, pages, home, XML feeds, etc.) and prints the response status, headers, and body to stdout. Both subcommands perform a mock HTTP round-trip through `server.Handler` without starting a server.

## Navigation Menu

The nav bar is built by `blog.GetMenu()`. Three sources, rendered in order:
1. `[[menu_links]]` entries in `config.toml` — static custom links (always shown inline).
2. `[[menu.pinned]]` entries — category links shown directly in the bar (no dropdown), sorted by `order`.
3. `[[menu.dropdowns]]` entries — a list of dropdown menus. Each dropdown entry specifies a `label` and a list of category references under `item` (each referencing `category` and an `order` hint).

`MenuLink` carries a `SubItems []MenuLink` field. When `SubItems` is non-empty the template renders a `<div class="nav-dropdown">` with a button trigger; when empty it renders a plain `<a>` tag. Dropdown CSS lives in `base.style.css` (`.nav-dropdown`, `.nav-dropdown-content`); the anthropic theme overrides colours in `anthropic.style.css`.

## Category Config Flags & Sub-categories

Each `[categories.<slug>]` block in `config.toml` supports:

| Flag | Type | Effect |
|------|------|--------|
| `blog_name` | string | Display name in nav and category header |
| `header_content` | string | Subtitle shown on category listing page |
| `folder` | string | Subfolder under `posts/` (defaults to slug) |
| `index` | bool | Include in homepage landing page index category cards |
| `menu` | bool | Add link to the nav bar |

### Parent and Sub-categories

MDBlog dynamically resolves parent/child category relationships by checking if a sub-category's folder starts with a parent category's folder prefix followed by a slash (e.g. `folder = "projects/android"` is resolved as a sub-category under parent category `projects` which has `folder = "projects"`).

- **Homepage Index Filtering**: Sub-categories should set `index = false` to hide them from the landing page cards.
- **Dynamic Render**: When a user visits the parent category page (e.g., `/?category=projects`), MDBlog populates `SubCategories` in `templateData` using `blog.GetSubCategories(parentSlug)`. The category template (`templates/category.html`) renders these sub-categories as cards at the top of the main listing, linking to `/?category=sub-category`.

## Landing Page, Category Page, Search, Feed, and Standalone Pages

The server handler has **six rendering paths**:

- **`/` with no query params** → Static landing page. Shows the blog header, optional `posts/index.md` blurb, then auto-generated category cards (only those with `index = true`). **No post scanning.**
- **`/?category=slug`** → Category post listing with pagination. If the category has sub-categories, it also renders them as card grids at the top.
- **`/?q=keyword`** or **`/?search=1`** → Standalone search results page using the JSON index.
- **`/feed.xml`** → Serves the pre-built RSS 2.0 feed from disk. Returns 404 when `feed.enabled = false`.
- **`/feed`** → Human-readable feed page listing all posts (date | category | title). Uses `blog.GetFeedPosts(maxItems)`.
- **`/page?slug=<slug>`** → Standalone page. Reads `<pages_dir>/<slug>.md`, renders it with `page.html` (no date, author, tags, or category breadcrumb).

To add a landing page blurb: create `posts/index.md` with Markdown content. Delete it to show only the category cards.

## Standalone Pages

Pages live in `PagesDir` (config key `pages_dir`, default `"pages"`). Any `<slug>.md` file there is served at `/page?slug=<slug>`.

- **`blog.GetPage(slug string) *Page`** — path-traversal-safe reader: validates that the resolved absolute path is a child of `PagesDir` before calling `os.ReadFile`. Returns `nil` for unknown slugs.
- **`templates/page.html`** — minimal template: just title + rendered body + a back-to-home link. No metadata section.
- Pages are **not** indexed by `buildindex`, not included in the RSS feed, and not scanned for search. Add nav links via `[[menu_links]]` in `config.toml`.

## Coding Conventions

- **Go 1.26+ required.** Use standard Go idioms; avoid unnecessary abstractions.
- **No ORMs, no frameworks** — standard library `net/http` + Goldmark + go-toml are the only non-AWS dependencies.
- **XSS prevention:** post content is rendered by Goldmark **without** `html.WithUnsafe()`. Template variables in `html/template` are auto-escaped. Raw HTML is only emitted via explicit `template.HTML` casts in trusted code paths.
- **Path traversal prevention:** `GetPostBySlug` and `GetPage` both validate that resolved file paths are within their respective directories (`PostsDir` / `PagesDir`). Any new file-reading code must do the same.
- **CSP:** configured via `config.toml` `[csp]` block. Do not add `unsafe-eval` without a documented security justification.
- **Gzip:** the server handler applies gzip compression. Any new HTTP entry point should do the same.
- **No globals:** pass `*config.Config` and `*blog.Blog` explicitly. Do not introduce package-level mutable state (except the `TemplateFS`/`AssetsFS` embed overrides).
- **Tests live in the same package** (white-box style). Run with `make test`.

## Adding a New Category

1. Create `posts/<folder-name>/` and add `.md` files.
2. Register the category in `config.toml`:
   ```toml
   [categories.my-category]
   blog_name      = "Display Name"
   header_content = "Subtitle shown on category index."
   folder         = "my-category"
   index          = false
   menu           = true
   ```
3. Run `make build-index` to regenerate the post metadata index.

## Post Metadata Index

Listing, pagination, and search are driven by a **build-time metadata index** (`posts/posts.index.json`). This avoids scanning and parsing all Markdown files on every request.

- **Generated by:** `make build-index` → `internal/buildindex.Build()`
- **Used by:** `blog.GetPosts()` (listing + pagination), `blog.SearchPosts()` (search), and `blog.GetPostBySlug()` (fallback slug→category resolution)
- **Not used for rendering:** individual post pages still parse the full Markdown body, but only for the single requested post
- **Fallback:** if the index is absent or corrupt, `GetPosts()` falls back to a live filesystem scan and logs a warning

### Index contents (per post)

| Field | Source |
|-------|--------|
| `slug` | derived from filename |
| `title` | front matter or filename |
| `date` | front matter or file mtime |
| `author` | front matter |
| `tags` | front matter |
| `description` | front matter |
| `excerpt` | `description` or plain-text strip of raw Markdown body |
| `category_slug` | parent folder slug |
| `source_path` | relative path to `.md` file |
| `filename` | basename of the `.md` file |

Full post bodies are **never rendered** during index generation (Goldmark is not called).

## Lambda Deployment Notes

- The Lambda binary is a statically linked Go binary (`CGO_ENABLED=0`). No runtime dependencies.
- `algnhsa` adapts API Gateway v1/v2 HTTP events into `http.Handler` calls transparently.
- The container filesystem is read-only. There is no file-based cache — the JSON index is baked into the image and read-only at runtime.
- **Recommended caching strategy:** place **CloudFront** in front of the Lambda function. Since posts change only on redeploy, a CloudFront TTL of hours or days is safe. Invalidate the distribution after `make docker-push`.

## What NOT to Do

- Do not add a database layer.
- Do not introduce a front-end build tool (webpack, vite, etc.).
- Do not set `html.WithUnsafe()` in the Goldmark configuration.
- Do not add `unsafe-eval` to the CSP without a documented security justification.
- Do not put new content in the root `posts/` dir (unless `show_uncategorized = true`) — use a category folder.
- Do not run the container as root or re-add dropped Linux capabilities without justification.
- **When modifying the Makefile**, always update `README.md` and `wiki/agents.md` to match.
- Do not add SEO-related code to `handler.go` — all JSON-LD helpers, sitemap structs, and robots/sitemap HTTP handlers live in `server/seo.go`.
- Do not add a new page type without also adding a corresponding `templates/<type>.html` and a route in `server/handler.go`.
- Do not index standalone pages (`pages/`) in the post metadata index — they are not posts.
- Do not routinely ingest or rewrite user-authored `posts/` content as part of wiki maintenance.

## Lessons Learned & Best Practices

### Adding a New Label

Any user-visible string in a template must go through the `[labels]` block in `config.toml`. The full checklist for adding a label:

1. Add the field to the `Labels` struct in `internal/config/config.go` with the correct `toml:` tag.
2. Add the key and default value to `config.toml` under `[labels]`.
3. Reference it in the template as `{{ .Config.Labels.FieldName }}`. Use `| defaultStr "fallback"` as a safety net.
4. Never hardcode user-visible strings directly in templates or Go handler code.

### Go Templates: `range` scope and `$`

Inside a `{{ range }}` block, `.` is rebound to the current iteration item. Accessing root template data (e.g. `Config`) requires the `$` prefix:

```html
{{ range .Categories }}
    {{ .BlogName }}                          {{/* CategoryInfo field — OK */}}
    {{ $.Config.Labels.PostsLabel }}         {{/* root data — must use $ */}}
{{ end }}
```

Forgetting `$` produces: `can't evaluate field Config in type blog.CategoryInfo`.

### CSS: Cards in a Grid — Shadow and Hover Pitfalls

When using `display: grid` with card elements that lift on hover (`transform: translateY`):

- **Shadow clipping:** adjacent cards in the same grid row render on top and clip the raised card's `box-shadow`. Fix: add `z-index: 1` to the `:hover` rule.
- **Underline on hover:** `<a>` cards inherit `a:hover { text-decoration: underline }` from the base theme. Fix: add `text-decoration: none` explicitly to the card's `:hover` rule.
- **Bottom border stripped:** if the base theme sets `.post-preview:last-child { border-bottom: none }` (to remove list separators), card-style themes must override that back to `border-bottom: 1px solid var(--color-card-border)` since cards use their full border for visual separation.

### CSS: Font Consistency in the Anthropic Theme

The anthropic theme uses two font stacks:
- `--body-font` (Georgia/serif) — prose content: post body, excerpts.
- `--ui-font` (system sans-serif) — all UI chrome: nav, meta, tags, tables, feed, cards.

The feed table is UI chrome, not prose — set `font-family: var(--ui-font)` on `.feed-table` itself rather than per-cell. The base theme anchors `.feed-table` to `--body-font` for consistency; the anthropic theme overrides it to `--ui-font`.

### Config Values Available in Templates

`templateData` exposes the full `*config.Config` as `.Config`. This means any config value — including structs like `.Config.Feed.MaxItems` — is accessible directly in templates without adding new handler fields. Check `internal/server/handler.go` → `templateData` struct before assuming a value isn't available.

### Custom Post Slugs and Dates

By default, posts use the file naming format `YYYY-MM-DD-slug.md` which results in a date-prefixed slug (e.g. `2026-06-03-my-app`). For showcasing products or projects where a clean, permanent URL is desired (e.g. `/post?slug=my-app`), you can name the file without the date prefix (e.g. `my-app.md`). In this case, you **must** specify the date in the post's YAML front matter (`date: 2026-06-03`) to ensure it indexes and displays with the correct date.
